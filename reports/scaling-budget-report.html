Deployment Shapes

1. Render Shape: Near-Term Production

Best for early public launch, low operational overhead, and a small team. This is the easiest next step from the current setup.

Cloudflare Pages Render Web Render Postgres Upstash Redis Optional Render worker

2. AWS Lightsail Shape: Medium-Term Growth

Best for a growing audience, better horizontal capacity, and more predictable scaling costs without the complexity of a full AWS microservice setup.

Cloudflare Pages 2-3 Lightsail app instances Lightsail load balancer Managed PostgreSQL Upstash Redis Cloudflare R2

3. AWS ECS / RDS Shape: Long-Term Scale

Best for higher reliability, multiple services, larger spikes, and more formal backend operations once the product outgrows simpler platforms.

Cloudflare Pages ECS / Fargate API service ECS worker service RDS PostgreSQL Redis ALB Cloudflare R2

Recommended Architecture for the Current Codebase

This codebase already has separate concerns for notifications, polls, live updates, response caching, email, push delivery, and scheduled ingestion. The most important scaling step is to stop letting the web API process own everything.

Edge and Frontend

  • Keep Cloudflare Pages for admin web and any future client web.
  • Serve static assets and frontend bundles through Cloudflare.
  • Store editorial and live-update media in Cloudflare R2 instead of pushing image load through the API.

API Layer

  • Run FastAPI as a stateless API service.
  • Once traffic grows, run 2 or more API instances behind a load balancer.
  • Do not let multiple API instances all own the scheduler or ingestion loop.

Worker Layer

  • Create a dedicated worker service for scheduled ingestion and future background jobs.
  • Move scheduler ownership out of the main API process before horizontal scaling.
  • Later extend the worker to handle email outbox and push-notification outbox delivery.

Database

  • Upgrade off free Postgres immediately.
  • Set explicit database connection pool sizing in production.
  • Later add PgBouncer or a database proxy once concurrency grows further.

Cache Layer

  • Use Redis for response caching.
  • Cache homepage and article-list reads for 15-60 seconds.
  • Cache active polls and live-update page reads for 5-30 seconds.
  • Invalidate cache when articles publish, polls receive votes, or live-update entries change.

Practical Recommendation

For this codebase, the best path is:

  1. Move immediately to a paid Render database and Redis-backed caching.
  2. Use Render Standard for the API once public traffic becomes real.
  3. Split scheduler and ingestion into a worker before adding multiple API instances.
  4. Move to AWS Lightsail only when the product clearly outgrows a single-instance Render shape.
  5. Move to ECS / RDS only when operational complexity is justified by clear usage and reliability needs.
