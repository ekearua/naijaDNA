Corrected Cost Logic

  • A cached setup is usually the same price or slightly more expensive as a monthly bill.
  • A cached setup usually serves more users before the next upgrade.
  • That means it is cheaper per 1,000 daily active users, even if the bill itself is not lower.

Platform Recommendation

Recommended now: Keep LiveKit

  • It is already integrated in both backend and Flutter.
  • The current stream feature is interactive, not just passive broadcast.
  • Changing providers now would create avoidable backend and client rewrite work.
  • LiveKit already fits the current room-token architecture and viewer join flow.

Recommended later: add Mux if streams become broadcast-heavy

  • LiveKit for host, backstage, or guest coordination.
  • Mux for large-scale viewer delivery.
  • Keep the rest of the application flow unchanged.
Recommendation in one line: keep LiveKit now, because it matches the existing implementation; only move to Mux or a LiveKit plus Mux hybrid when the audience becomes mostly passive and stream viewer counts materially outgrow the economics of interactive WebRTC delivery.

Architecture Notes For This Codebase

  • Keep frontend delivery on Cloudflare Pages.
  • Move scheduled ingestion and the scheduler out of the API process before adding multiple API instances.
  • Keep Redis as a shared response cache for homepage reads, poll reads, category reads, live-update reads, and stream-related read endpoints where appropriate.
  • Upgrade off free Postgres immediately if the app is meant to serve real users.
  • Move media to Cloudflare R2 so article, live-update, and stream-related images do not rely on app-disk storage.
